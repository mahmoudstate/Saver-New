# CLAUDE.md — قواعد ثابتة للمشروع

## 1. تنظيم وتقسيم الملفات (Code Organization)
- أي ملف كومبوننت يتجاوز 300 سطر لازم يتقسم لأجزاء أصغر، كل جزء مسؤول عن حاجة واحدة بس (Single Responsibility).
- افصل بين:
  - **UI Components**: أجزاء الواجهة الواضحة (Header, Form, List, Card...) كل واحد في ملف مستقل.
  - **Hooks / Logic**: أي state أو API calls تتحط في custom hook منفصل (مثال: `useUserData.js`).
  - **Utils / Helpers**: أي دوال حسابية أو تنسيق بيانات في ملف utils منفصل.
- الملف الرئيسي للشاشة يفضل صغير ومجرد "تجميع" للكومبوننتس دي.
- قبل أي تقسيم لملف موجود: اعرض الخطة المقترحة للتقسيم أول حاجة، ومتنفذش غير بعد موافقة صريحة.
- لما تنشئ ميزة جديدة، اكتبها من الأول بنفس الفلسفة دي (ملفات صغيرة منفصلة) بدل ما تتجمع كلها في ملف واحد.

## 2. أسلوب الرد (Communication Style)
- ردودك دايمًا مختصرة جدًا. ممنوع الشرح الزائد أو السرد التفصيلي لما عملته.
- ممنوع تكتب فقرات طويلة قبل أو بعد التنفيذ بتشرح فيها كل خطوة بالتفصيل.
- بعد التنفيذ: سطر أو اتنين بس يقولوا اتعمل ايه، مفيش تكرار أو احتفال بالنجاح.

## 3. قبل تنفيذ أي طلب تعديل
لازم ترد أولاً بخطة تنفيذ مختصرة (Plan) قبل ما تكتب أي كود، تتضمن:
1. تأكيد مختصر إنك فاهم المطلوب (سطر واحد).
2. خطوات التنفيذ في نقاط مختصرة (بدون شرح زايد).
3. تقدير: هل التعديل **بسيط/سريع** ولا **متوسط** ولا **كبير وهياخد وقت وتوكنز كتير**.
4. لو التعديل كبير أو هيحتاج فتح ملفات كتير، نبهني قبل ما تكمل واسأل تأكيد.

متبدأش تنفيذ فعلي غير بعد ما أرد بـ "تمام" أو "ابدأ" أو ما يفيد الموافقة.

## 4. قواعد عامة
- متفتحش/متقراش ملفات مش مرتبطة بالتعديل المطلوب.
- لو الطلب غامض، اسأل سؤال توضيحي واحد بس مختصر بدل ما تفترض.
